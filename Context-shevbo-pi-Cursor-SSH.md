# Контекст для Cursor: `ssh shevbo-pi`

Краткая справка для агента/разработчика, у которого в окружении уже настроен **`ssh shevbo-pi`** (ключи, `~/.ssh/config`).

### SSH с ноутбука: не используйте `10.66.0.2` как HostName

**`10.66.0.2`** — это адрес **Raspberry Pi внутри WireGuard**, а не обязательно интерфейс, на котором слушает **sshd**. С домашнего ПК команда **`ssh shevbo-pi`** должна идти на **LAN** Pi (часто **`192.168.1.x`**) или на тот хост, куда у вас реально проброшен SSH.

Если в **`~/.ssh/config`** для **`Host shevbo-pi`** стоит **`HostName 10.66.0.2`**, получите **`Connection timed out`**, пока на **вашем** ПК нет маршрута в **`10.66.0.0/24`** через свой WG (обычно его нет).

**Что сделать:** в блоке **`Host shevbo-pi`** верните **`HostName`** на **IP в локальной сети** (как раньше, напр. **`192.168.1.106`**). Вход **с облака** на Pi по **`10.66.0.2`** оставьте **отдельным** блоком, например **`Host shevbo-pi-from-cloud`** с `HostName 10.66.0.2`, и используйте его **только** с **`shevbo-cloud`**.

---

## 1. Что это за хост

- **`shevbo-pi`** — Raspberry Pi (нода), в сценарии OpenClaw часто **node** за шлюзом на **`shevbo-cloud`**.
- Доступ с вашей машины: обычно **локальная сеть** или **тот же SSH config**, что и для остальной инфраструктуры; не путать с путём **shevbo-cloud → Pi по VPN** (`10.66.0.2`), который использует другой ключ и видим только с облака.

---

## 2. Сеть и WireGuard (связь с облаком)

- На **shevbo-cloud** интерфейс **`wg0`**, адрес в туннеле часто **`10.66.0.3/32`**, подсеть **`10.66.0.0/24`**.
- На **Pi** в SSH-конфиге **с облака** Pi задаётся как **`HostName 10.66.0.2`**, пользователь **`shevbo`**, отдельный ключ (например `pi_from_cloud_ed25519`) — это **не** обязательно тот же вход, что `ssh shevbo-pi` с ноутбука.
- Если на облаке **`wg show`** показывает **старый `latest handshake`** (много часов), а **`ping 10.66.0.2`** с облака не идёт — Pi **не в туннеле** (питание, WAN, UDP до WG, конфиг WG).

**Диагностика на консоли Pi:**  
`bash scripts/pi/shevbo-pi-wg-cloud-check.sh` (из клона репо на Pi или скопируйте файл в `~/bin`) — ping интернета, `wg show`, `wg-quick@wg0`, ping **`10.66.0.3`**, опционально **`--restart`**. Переменные: **`WG_IF`**, **`CLOUD_WG_IP`** при нестандартных адресах.

## 4. Практика для Cursor с SSH на Pi

1. Сначала: **`hostname`**, **`uptime`**, **`ip -br a`**, при проблемах с облаком — скрипт из п. 2.  
2. Не коммитить и не вставлять в чаты **секреты** (ключи WG, токены, содержимое `environment.env` прокси).  
3. Команды длинные/разовые — выполнять через SSH самому агенту, а не только выдавать пользователю.  
4. Если задача про **OpenClaw на ноде** — смотреть, где установлен бинарь/юниты (`systemctl --user` vs системные), путь состояния (`~/.openclaw` и т.д. на том пользователе, под которым заходите).

---

## 5. Однострочник проверки с машины с Cursor

```bash
ssh shevbo-pi 'hostname; uptime; ip -br a | head -20'
```

После восстановления связи с облаком имеет смысл с **облака** проверить: **`ping 10.66.0.2`**, **`ssh shevbo-pi 'uptime'`** (если в cloud config так зовётся Pi), **`sudo wg show`** — свежее **handshake**.
